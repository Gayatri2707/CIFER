// The Problem & Solution

Why CIFER exists

Traditional file sharing is broken. Emails travel in plain text. Cloud providers can access your files. Shared passwords leak. There's no real verification that the right person received your data.

CIFER combines two techniques: AES-256 encryption makes the file mathematically unreadable, and steganography bundles it inside a cute nature image. Receivers must verify identity via a time-limited 6-digit OTP sent to their Gmail.

// Under the Hood

Technical Capabilities

Four cryptographic layers working in concert to protect your most sensitive files.

01 · AES-256-CBC

Military-Grade File Encryption

Uses the Advanced Encryption Standard with a 256-bit key and Cipher Block Chaining mode. A fresh random 16-byte IV is generated for every upload to guarantee uniqueness. Your file becomes completely unreadable without the correct key.

02 · STEGANOGRAPHY

Random Cover Images

At encryption time, the server randomly selects one of 10 cute nature themes — flowers, animals, landscapes. The encrypted payload hides invisibly in the pixel data. No one can tell the image contains a secret.

03 · GMAIL OTP

Identity Verification

Users receive a 6-digit OTP via secure SMTP to their Gmail. OTPs are hashed with SHA-256 in the database — never stored in plain text. Time-limited and single-use for maximum security.

04 · SHA-256 INTEGRITY

Tamper Detection

A SHA-256 hash is computed for every file at upload time. On decryption, the hash is re-computed and compared — any modification, even a single bit, is immediately detected.

STAGE 01

File Ingestion

Raw binary streamed to Flask backend, up to 50 MB

Stored temporarily in server memory for hashing & encryption — never persisted to disk unencrypted

STAGE 02

SHA-256 Hash

Tamper-detection fingerprint computed before encryption

hashlib.sha256(file_bytes) — stored hashed in the database for integrity check on decrypt

STAGE 03

AES-256-CBC

256-bit key + fresh random 16-byte IV encrypts the file

PyCryptodome AES.new(key, AES.MODE_CBC, iv) — new IV generated per upload for uniqueness

STAGE 04

Steganography

Ciphertext embedded inside a random nature cover image

LSB encoding hides the payload in pixel RGB channels — visually identical to the original image

STAGE 05

OTP Delivery

SHA-256 hashed 6-digit OTP emailed to each receiver via Gmail SMTP

smtplib sends OTP — only its SHA-256 hash is stored in DB, never the plain text OTP
